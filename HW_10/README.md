# NetworkLab — учебный TCP клиент-сервер на .NET 10

Учебный проект к практическому заданию №10 «Основы сетевого программирования в .NET».
Реализует базовую клиент-серверную архитектуру поверх TCP-сокетов с надёжной передачей сообщений и измерением производительности.

---

## 📦 Требования

- **.NET SDK 10.0.x** (проект таргетит `net10.0`).
- Любая ОС, где работает .NET 10 (Windows / Linux / macOS).
- Свободный TCP-порт `8888` на `127.0.0.1`.

Проверить установленный SDK:

```bash
dotnet --list-sdks
```

---

## 🚀 Сборка и запуск

```bash
# Из папки проекта
dotnet build
dotnet run -c Release
```

Программа сама поднимет сервер, прогонит все тесты и распечатает итоговую сводку, после чего корректно завершится. **Интерактивного меню нет** — это автоматический демо-прогон.

## 🧭 Логика, объяснённая по шагам

### 1. Формат сообщения и почему нужен префикс длины

TCP — это **поток байтов**, а не «коробочки» по одному сообщению. Если просто слать байты подряд, принимающая сторона не сможет понять, где заканчивается одно сообщение и начинается следующее.

Решение: **перед каждым сообщением кладём 4 байта длины**.

```
[4 байта длины N] [N байт тела сообщения]
```

Тело сообщения мы записываем `BinaryWriter`-ом в фиксированном порядке:

```
[16 байт]  MessageId  (Guid)
[string]   MessageType    -- BinaryWriter.Write(string) сам пишет длину + UTF-8
[8 байт]   Timestamp      -- DateTime.ToBinary() -> long
[string]   SenderId
[4 байта]  PayloadLength
[N байт]   Payload
```

Это лежит в `MessageProtocol.Serialize` и `MessageProtocol.Deserialize`.

### 2. Частичные чтения

Когда мы делаем `stream.ReadAsync(buffer, ...)`, нам могут вернуть **меньше байт, чем мы просили** — это нормальное поведение TCP. Поэтому всегда читаем в цикле, пока не наберём нужное количество. Делает это метод `ReadExactlyAsync` в `MessageProtocol`:

```csharp
while (totalRead < count)
{
    var bytesRead = await stream.ReadAsync(buffer.AsMemory(totalRead, count - totalRead), ct);
    if (bytesRead == 0) { /* поток закрылся */ }
    totalRead += bytesRead;
}
```

`ReadMessageAsync` использует этот метод дважды: сначала читает ровно 4 байта длины, потом ровно `N` байт тела.

### 3. Один универсальный класс `TcpClientWrapper`

Класс отвечает за **одно** TCP-соединение и имеет два конструктора:

- `new TcpClientWrapper(host, port)` — для клиента, который потом вызывает `ConnectAsync()`.
- `new TcpClientWrapper(id, acceptedClient)` — для сервера, который только что принял подключение.

После того как соединение готово, вызывается `StartReceiving()` — фоновая задача в цикле читает сообщения и поднимает событие `MessageReceived`. На клиенте это происходит автоматически после `ConnectAsync`, на сервере — сразу после оборачивания принятого подключения.

Запись защищена `SemaphoreSlim _writeLock` — два потока не могут одновременно писать в один сокет (иначе length-prefix и тело перемешаются).

### 4. Сервер

`TcpServer` поднимает `TcpListener` на `127.0.0.1` и запускает фоновый цикл приёма подключений (`AcceptLoopAsync`). Для каждого нового клиента:

1. Берёт «слот» у `SemaphoreSlim _connectionLimiter` (ограничение на одновременные подключения — рекомендация задания).
2. Создаёт `TcpClientWrapper`, кладёт его в `ConcurrentDictionary<string, TcpClientWrapper> ConnectedClients`.
3. Подписывается на события wrapper'а (`MessageReceived`, `Disconnected`) и пробрасывает их наружу.
4. Запускает фоновый приём.

Когда клиент отключается, сервер чистит запись в словаре и освобождает слот у семафора.

`BroadcastMessageAsync` рассылает сообщение всем; `SendMessageToClientAsync` — конкретному по Id.

### 5. ACK-логика в `Program.cs`

Чтобы можно было реально измерять надёжность и latency, в `Program.cs` сервер подписан на `MessageReceived`. Когда приходит сообщение типа `"DATA"`, сервер отправляет в ответ `"ACK"`. Чтобы бенчмарк latency мог сопоставить запрос с ответом, **в `SenderId` ACK мы кладём исходный `MessageId`** в виде строки.

Это «прикладная» договорённость поверх протокола — сам `MessageProtocol` ничего об ACK не знает, что и правильно: сетевой слой остаётся универсальным.

### 6. Бенчмарки

| Метод                                       | Что меряет                                                                                                                                                 |
| ------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `BenchmarkSingleClientAsync(N, size)`       | Сколько занимает отправка `N` сообщений размера `size` одним клиентом + сколько ACK успело вернуться. Возвращает: время, отправлено, получено, **МБ/сек**. |
| `BenchmarkMultipleClientsAsync(K, N, size)` | То же, но параллельно `K` клиентов. Возвращает: время, всего отправлено, всего ACK.                                                                        |
| `BenchmarkThroughputAsync(seconds, size)`   | За `seconds` шлёт максимально быстро. Возвращает: байт отправлено, байт получено (ACK), **МБ/сек**.                                                        |
| `CompareLatencyAsync(iterations)`           | Round-trip: для каждой итерации меряет время от `SendMessageAsync` до получения ACK с тем же `MessageId`. Возвращает: среднее, мин, макс (мс).             |

Везде используем `Stopwatch` для высокоточного замера времени.

## ✅ Как решение соответствует запретам задания

| Запрет / требование                               | Где выполнено                                                                                                                 |
| ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| ❌ Не использовать SignalR / gRPC / WCF           | Используются только `System.Net.Sockets.TcpListener` и `System.Net.Sockets.TcpClient`.                                        |
| ❌ Не использовать `HttpClient`                   | Нет ни одного HTTP-клиента.                                                                                                   |
| ✅ Использовать **префикс длины**                 | `MessageProtocol.WriteMessageAsync` пишет 4 байта длины перед телом.                                                          |
| ✅ Обрабатывать **частичные чтения**              | `MessageProtocol.ReadExactlyAsync` читает в цикле, пока не наберёт `count` байт.                                              |
| ✅ Обработка исключений                           | `try/catch` в `AcceptLoopAsync`, `ReceiveLoopAsync`, `SafeSendAsync`, `ConnectAsync` (включая `TimeoutException`).            |
| ❌ Никаких глобальных переменных                  | Состояние подключений — в `ConnectedClients` сервера; никаких `static` контейнеров на верхнем уровне.                         |
| ✅ Освобождение ресурсов                          | `TcpClientWrapper` и `TcpServer` реализуют `IAsyncDisposable`; в `Program.cs` используются `await using` и явный `StopAsync`. |
| ✅ `BinaryReader` / `BinaryWriter`                | Используются в `Serialize` / `Deserialize`.                                                                                   |
| ✅ `CancellationToken`                            | Прокинут везде, где есть асинхронные циклы (`AcceptLoopAsync`, `ReceiveLoopAsync`, бенчмарки).                                |
| ✅ `Stopwatch` для измерений                      | Во всех методах `NetworkBenchmark`.                                                                                           |
| ✅ Асинхронные методы (`ReadAsync`, `WriteAsync`) | Везде, где работаем с потоком.                                                                                                |
| ✅ `SemaphoreSlim`                                | (1) ограничение одновременных подключений на сервере, (2) сериализация записи в каждом wrapper'е.                             |
| ✅ `Encoding.UTF8`                                | В `BinaryReader/Writer` и при формировании тестовых payload-ов.                                                               |

## 🧠 Если будут спрашивать на защите

Вот короткие ответы на типичные вопросы:

- **Почему вы пишете 4 байта длины перед сообщением?** Потому что TCP — это поток, без границ. Префикс длины позволяет приёмнику точно знать, сколько байт читать.
- **Что такое частичное чтение?** `Stream.ReadAsync` имеет право вернуть меньше байт, чем запросили. Поэтому мы всегда читаем в цикле, пока не наберём нужное количество.
- **Зачем `SemaphoreSlim` в `TcpClientWrapper`?** Чтобы две параллельные записи в один сокет не «перемешали» байты — иначе length-prefix одного сообщения окажется в середине тела другого.
- **Почему класс назван `TcpClientWrapper`, а не `TcpClient`?** Чтобы не конфликтовать с системным `System.Net.Sockets.TcpClient`. Наш wrapper — это надстройка над ним.
- **Как сервер понимает, что клиент отключился?** `ReadMessageAsync` возвращает `null` или бросает исключение, фоновый цикл выходит, поднимается событие `Disconnected`, сервер чистит словарь и освобождает слот у `SemaphoreSlim`.
- **Что делать, чтобы добавить ещё один тип сообщения?** Достаточно ввести новое значение в `MessageType` (например, `"PING"`) и обработать его в подписчиках. Сам формат сериализации менять не нужно.
