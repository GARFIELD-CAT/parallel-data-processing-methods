# Практическое задание 9: TPL Dataflow

Учебный проект по теме «асинхронные пайплайны обработки потоковых данных» на .NET 10 / C#.

## О чём проект

Программа моделирует обработку котировок биржевых терминалов с помощью библиотеки
**TPL Dataflow** (`System.Threading.Tasks.Dataflow`). Реализованы:

- линейный пайплайн (BufferBlock → TransformBlock → ActionBlock) с CPU-bound
  обработкой и параллелизмом;
- сложный пайплайн с ветвлением по предикату и общим приёмником;
- broadcast-пайплайн (одно сообщение → нескольким получателям);
- пайплайн с ограничением буфера (`BoundedCapacity`) и backpressure;
- «приоритезированный» пайплайн с двумя входными буферами;
- пайплайн обработки котировок: фильтрация → SMA → торговый сигнал → «БД»;
- бенчмарк, сравнивающий Dataflow с последовательной обработкой.

## Требования

- .NET SDK **10.0** или совместимая версия.
- Любая OS (Windows / Linux / macOS).

## Как собрать и запустить

В корне проекта (там, где лежит `HW_9.csproj`):

```bash
dotnet restore
dotnet build -c Release
dotnet run -c Release
```

> Запускайте в конфигурации **Release**: обработка одного сообщения сделана
> намеренно «тяжёлой» (CPU-bound), чтобы продемонстрировать выигрыш Dataflow,
> и в Debug замеры будут неинформативны.

## Краткое объяснение логики

Главная идея TPL Dataflow — это **блоки**, которые получают сообщения, обрабатывают их
и передают дальше. Блоки соединяются методом `LinkTo(...)`, образуя пайплайн.

Три основных типа блоков:

- `BufferBlock<T>` — буфер: принимает сообщения и отдаёт следующему блоку.
- `TransformBlock<TIn, TOut>` — преобразует сообщения (как `Select` в LINQ, но потоково).
- `ActionBlock<T>` — «терминал» пайплайна: что-то делает с сообщением.

Чтобы пайплайн корректно завершился, нужно:

1. вызвать `Complete()` на входном блоке (сигнал «больше сообщений не будет»);
2. дождаться `Completion` последнего блока (`.Wait()`).

Опция `PropagateCompletion = true` распространяет завершение по цепочке автоматически.
**Внимание:** при ветвлениях это приводит к ошибкам (две ветки закрывают общий приёмник),
поэтому в сложном пайплайне завершение собирается вручную через локальную `async`-функцию
(это оркестрация завершения, а не распараллеливание данных).

## Обработка ошибок

Тело каждого блока, выполняющего нетривиальную работу, обёрнуто в `try/catch`:
сбойное сообщение не роняет весь пайплайн. В торговом пайплайне ошибки считаются
отдельным счётчиком (`GetErrorCount`). Дополнительно класс `ErrorHandlingBlock`
демонстрирует перехват исключения внутри блока и продолжение работы, а `PostWithRetry`
повторяет отправку при переполнении блока.

## Производительность

Обработка одного сообщения (`DataflowPipeline.HeavyTransform`) — это CPU-bound цикл,
одинаковый для Dataflow-пайплайна и для последовательного бенчмарка (честное сравнение).
Простой пайплайн настроен на `MaxDegreeOfParallelism = ProcessorCount`, поэтому
раскидывает тяжёлую работу по ядрам и обгоняет однопоточную обработку.

> На тривиальной обработке (например, `x * 2`) Dataflow, наоборот, медленнее
> последовательного кода — накладные расходы на передачу сообщений между блоками
> превышают пользу. Преимущество проявляется именно на тяжёлой нагрузке.
