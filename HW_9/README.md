# Практическое задание 9: TPL Dataflow

Учебный проект по теме «асинхронные пайплайны обработки потоковых данных» на .NET 10 / C#.

## О чём проект

Программа моделирует обработку котировок биржевых терминалов с помощью библиотеки
**TPL Dataflow** (`System.Threading.Tasks.Dataflow`). Реализованы:

- линейный пайплайн (BufferBlock → TransformBlock → ActionBlock);
- сложный пайплайн с ветвлением по предикату и общим приёмником;
- broadcast-пайплайн (одно сообщение → нескольким получателям);
- пайплайн с ограничением буфера (`BoundedCapacity`);
- «приоритезированный» пайплайн с двумя входными буферами;
- пайплайн обработки котировок: фильтрация → SMA → торговый сигнал → «БД»;
- бенчмарк, сравнивающий Dataflow с последовательной обработкой.

## Требования

- .NET SDK **10.0.201** или совместимая версия .NET 10.
- Любая OS (Windows / Linux / macOS).

## Краткое объяснение логики (для новичка)

Главная идея TPL Dataflow — это **блоки**, которые получают сообщения, обрабатывают их
и передают дальше. Блоки соединяются методом `LinkTo(...)`, образуя пайплайн.

Три основных типа блоков:

- `BufferBlock<T>` — просто буфер: принимает сообщения и отдаёт следующему блоку.
- `TransformBlock<TIn, TOut>` — преобразует сообщения (как `Select` в LINQ, но потоково).
- `ActionBlock<T>` — «терминал» пайплайна: что-то делает с сообщением (сохраняет, логирует).

Чтобы пайплайн корректно завершился, нужно:

1. Вызвать `Complete()` на входном блоке (сигнал «больше сообщений не будет»);
2. Дождаться `Completion` последнего блока (`.Wait()` или `await`).

Если поставить опцию `PropagateCompletion = true` при связывании, то завершение
будет распространяться по цепочке автоматически. **Внимание:** при ветвлениях
это может привести к ошибкам — поэтому для сложных схем мы управляем завершением вручную.
