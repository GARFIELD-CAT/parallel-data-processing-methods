# Практическое задание 5 — Безопасные коллекции

## Описание

Реализована тестовая система, демонстрирующая использование потокобезопасных коллекций из `System.Collections.Concurrent`:

- `ConcurrentDictionary` — в `ConcurrentLibraryCatalog`
- `BlockingCollection` — в `TaskQueueManager`
- `ConcurrentDictionary` + `ConcurrentBag` — в `ConcurrentCache`

## Ключевые положения

- Запрещено использовать `lock`, `Monitor`, `Mutex`, `Semaphore`, `ReaderWriterLockSlim` для синхронизации доступа к коллекциям. В реализации синхронизации с обычным `Dictionary` используется локальный `lock` только в методе сравнения для демонстрации разницы (это часть задания).
- Используются атомарные методы коллекций: `TryAdd`, `TryRemove`, `TryUpdate`, `GetOrAdd`, `AddOrUpdate`.
- Для очереди задач применяется `BlockingCollection` с ограниченной емкостью и `GetConsumingEnumerable()` для потребления.
- Stopwatch используется для точных замеров.
- `Random` с фиксированным seed (42) для воспроизводимости.

## Структура файлов

- Program.cs — точка входа, подготовка тестовых данных, запуск тестов и вывод сводной статистики.
- ConcurrentLibraryCatalog.cs — потокобезопасный каталог книг на основе `ConcurrentDictionary`.
- TaskQueueManager.cs — менеджер очереди задач на основе `BlockingCollection`.
- ConcurrentCache.cs — простой кэш на `ConcurrentDictionary<string, ConcurrentBag<CacheItem>>`.
- CollectionBenchmark.cs — бенчмарки для `ConcurrentDictionary`, `BlockingCollection`, и синхронизированного `Dictionary`.
- README.md — этот файл.

## Как запускать

1. Убедитесь, что установлен .NET SDK 10.0 (в задании указана версия).
2. Создайте проект (или поместите файлы в существующий консольный проект).
3. Выполните:
   dotnet run

## Что делает программа

- Генерирует 1000 книг, 1000 задач и 500 элементов кэша (фикcированный seed).
- Запускает бенчмарки:
  - ConcurrentDictionary с конкурентными операциями.
  - Синхронизированный Dictionary (через lock) для сравнения.
  - BlockingCollection с 10 обработчиками задач.
  - ConcurrentCache тесты.
- Выводит время, число успешных операций, производительность (операций/сек) и простую проверку целостности.

## Важные моменты

- Почему `ConcurrentDictionary` быстрее в конкурентном доступе: внутренние структуры и атомарные операции уменьшают конкуренцию по сравнению с глобальными lock.
- Использование `GetOrAdd` и `AddOrUpdate` для безопасной ленивой и атомарной инициализации/обновления.
- `BlockingCollection.GetConsumingEnumerable()` блокирует и корректно заканчивает перебор после `CompleteAdding()` — это упрощает работу с consumer-threads.
- В `ConcurrentCache` используется `ConcurrentBag` для хранения версий значений; чтение избегает удаления элементов, поэтому простая проверка `IsExpired` демонстрирует принцип TTL, но для production можно добавлять фоновые очистки.

## Ограничения и возможные улучшения

- В `ConcurrentCache` элементы не удаляются автоматически при истечении срока — можно добавить периодический GC.
- В `TaskQueueManager` демонстрационно подавляются исключения в задачах — в реальном приложении логирование и обработка ошибок обязательны.
- Benchmark упрощён: для более точных измерений можно варьировать число потоков, размер задач и нагрузку.
