Кратко  
Показаны три подхода к синхронизации операций с банковским счётом: без синхронизации, с `lock` и с `Monitor`. Демонстрируются гонки данных и сценарий deadlock с решением через единообразный порядок захвата блокировок. Замеры производительности выполняются с помощью `Stopwatch`. Тесты воспроизводимы — `Random` с seed = 42.

## Файлы проекта

- `Program.cs` — точка входа, генерация транзакций, запуск бенчмарков и демонстрация deadlock/решения.
- `BankAccount.cs` — реализация баланса:
  - базовые методы (без синхронизации),
  - методы с `lock`,
  - методы с `Monitor` (с корректным try/finally),
  - методы с таймаутом (используют `Monitor.TryEnter` и возвращают `bool`).
- `TransactionProcessor.cs` — параллельная обработка транзакций; демонстрация deadlock и решение через единообразный порядок захвата (по Id).
- `SynchronizationBenchmark.cs` — замеры времени для всех подходов и их сравнение.
- `README.md` — этот файл.

## Что проверяется

- Корректность итогового баланса при параллельной обработке транзакций.
- Наличие/отсутствие гонок данных (без синхронизации — вероятны).
- Deadlock при наивном захвате блокировок и его устранение через упорядоченный захват.
- Временные накладные расходы при использовании `lock` и `Monitor`.

## Основные реализации и решения

- Для предотвращения deadlock в методах перевода (`TransferWithLock`, `TransferWithMonitor`) блокировки захватываются в унифицированном порядке по уникальному `Id` аккаунта.
- Для таймаутов используются `Monitor.TryEnter(timeoutMs)`; методы возвращают `true` при успешном захвате и выполнении операции, `false` при таймауте.
- Все операции с синхронизацией освобождают блокировки в блоке `finally` (обязательно).

## Примеры вывода (образец)

- Сравнение подходов:
  - Без синхронизации: [время] мс, результат: корректный/некорректный
  - С использованием lock: [время] мс, результат: корректный
  - С использованием Monitor: [время] мс, результат: корректный
- Итоговая сводка содержит время, итоговый баланс, корректность и накладные расходы в процентах.

## Ограничения и соблюдение условий задания

- Не используются: `Semaphore`, `Mutex`, `ReaderWriterLock` и др.
- Не используются: `System.Threading.Tasks` для синхронизации, `async/await` в методах синхронизации.
- Нет глобальных переменных для синхронизации.
- Исключения при захвате блокировок не игнорируются; блокировки всегда освобождаются в `finally`.

## Пояснения для сдачи преподавателю

- Поясните, что без синхронизации на конкурентных модификациях `decimal`-баланса возникают состояния гонки (interleaving операций), приводящие к неверным суммам.
- Продемонстрируйте deadlock‑сценарий (два потока, обратный порядок захвата) и объясните, как упорядочивание захватов по `Id` устраняет проблему.
- Обсудите замеры: `lock` и `Monitor` обеспечивают корректность, но добавляют накладные расходы (зависят от нагрузки/CPU).

## Запуск тестов/отладки

- Для детальной отладки уменьшите число транзакций и включите логирование операций или счётчик успешных операций.
- Для воспроизводимости всегда используйте фиксированный seed (`42`), как в проекте.
