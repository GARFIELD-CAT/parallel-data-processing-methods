# Практическое задание 8 — PLINQ: параллельные запросы к данным

Учебный проект демонстрирует, как с помощью **PLINQ** (Parallel LINQ)
ускорить обработку больших наборов финансовых данных по сравнению
с обычным «последовательным» LINQ.

## Требования

- **.NET SDK 10.0.201** (или совместимая версия .NET 10)
- Любая ОС, поддерживаемая .NET (Windows / Linux / macOS)

## Сборка и запуск

```bash
# Из корня проекта
dotnet new console -n PLINQDemo      # если проект ещё не создан
# Затем заменить файлы Program.cs / добавить остальные .cs
dotnet build
dotnet run -c Release                # обязательно в Release — иначе замеры неточные
```

> ⚠️ Для корректных замеров производительности запускайте проект
> в конфигурации **Release**, без отладчика (`Ctrl+F5` в Visual Studio).

## Что демонстрирует программа

1. **Фильтрация** записей по диапазону цены — простой `Where`.
2. **Агрегация** — `Sum / Average / Min / Max / Count`.
3. **Трансформация** — расчёт **простой скользящей средней** (окно 10).
   Здесь критически важно сохранить **порядок** элементов, поэтому
   в параллельной версии используется `AsOrdered()`.
4. **Соединение** двух наборов по `Symbol` (`Join`).
5. **Поиск оптимальной степени параллелизма** для разных нагрузок
   (1, 2, 4, 8, 16 потоков).
6. **Пресеты `QueryOptimizer`** для трёх типичных сценариев:
   - большие данные (`FullyBuffered`, максимум ядер),
   - зависящие от порядка (`AsOrdered`),
   - низкая задержка (`NotBuffered`, меньше потоков).

## Пошаговая логика (для устного объяснения)

1. **Генерация**: `DataGenerator.GenerateMarketData(n)` создаёт `n` записей.
   Используется `Random(42)` — фиксированный seed нужен,
   чтобы при каждом запуске получались одинаковые данные
   и сравнение производительности было честным.

2. **Последовательный анализ**: обычный LINQ работает в **одном потоке**.
   Например, `data.Where(...)` пробегает коллекцию подряд.

3. **Параллельный анализ**: всё начинается с метода **`AsParallel()`**.
   Он превращает `IEnumerable<T>` в `ParallelQuery<T>`,
   после чего PLINQ автоматически:
   - разбивает данные на части,
   - распределяет их между потоками (по числу ядер CPU),
   - вычисляет каждую часть отдельно,
   - объединяет результат.

4. **Когда важен порядок** (скользящая средняя зависит от позиции),
   мы добавляем **`AsOrdered()`** — это гарантия, что итоговая
   последовательность будет в том же порядке, что и входная.

5. **Контроль параллелизма** — `WithDegreeOfParallelism(n)`:
   полезно, когда мы не хотим занимать все ядра CPU
   (например, на сервере, где параллельно работают другие задачи).

6. **`WithMergeOptions`**: настраивает, как PLINQ «склеивает» результаты.
   - `NotBuffered` — отдаёт элементы сразу (низкая задержка).
   - `AutoBuffered` — компромисс (по умолчанию).
   - `FullyBuffered` — собирает всё, потом отдаёт (макс. пропускная способность).

7. **`Stopwatch`** замеряет время. Перед основным замером — **прогрев**:
   первый запуск всегда медленнее из-за JIT-компиляции.

8. **Проверка корректности**: для каждой операции мы запускаем
   последовательную и параллельную версии и сравниваем результаты.
   Для фильтрации сравниваем как множества (порядок может отличаться),
   для агрегации — значения, для трансформации — поэлементно.

## Соблюдение запретов из задания

| Запрет                                               | Как соблюдён                                                           |
| ---------------------------------------------------- | ---------------------------------------------------------------------- |
| Нельзя использовать `Task`, `Thread`, `Parallel.For` | Используется только `AsParallel()` и операторы PLINQ.                  |
| Нельзя использовать `async/await`                    | Все методы синхронные.                                                 |

| Запрет на внешние библиотеки                         | Использован только базовый .NET (`System.Linq`, `System.Diagnostics`). |

## Что важно понимать про производительность PLINQ

- На **малых данных** (10 000 записей) PLINQ часто **медленнее**
  обычного LINQ — накладные расходы на запуск потоков и слияние
  результатов перевешивают пользу от параллелизма.
- Эффект становится виден на **сотнях тысяч и миллионах** записей.
- Сложность операции тоже важна: чем «тяжелее» каждый элемент,
  тем сильнее выигрыш от параллелизма.
